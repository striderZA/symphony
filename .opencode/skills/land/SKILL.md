# Land

## Goals

- Ensure the PR is conflict-free with main.
- Keep CI green and fix failures when they occur.
- Squash-merge the PR once checks pass.
- Do not yield to the user until the PR is merged; keep the watcher loop running
  unless blocked.

## Preconditions

- `gh` CLI is authenticated.
- You are on the PR branch with a clean working tree.

## Steps

1. Locate the PR for the current branch.
2. Confirm tests are green locally before any push (`python -m pytest python/tests/ -v`).
3. If the working tree has uncommitted changes, commit with the `commit` skill
   and push with the `push` skill before proceeding.
4. Check mergeability and conflicts against main.
5. If conflicts exist, use the `pull` skill to fetch/merge `origin/main` and
   resolve conflicts, then use the `push` skill to publish the updated branch.
6. Watch checks until complete.
7. If checks fail, pull logs, fix the issue, commit with the `commit` skill,
   push with the `push` skill, and re-run checks.
8. When all checks are green, squash-merge using the PR title/body for the
   merge subject/body.

## Commands

```
# Ensure branch and PR context
branch=$(git branch --show-current)
pr_number=$(gh pr view --json number -q .number)
pr_title=$(gh pr view --json title -q .title)
pr_body=$(gh pr view --json body -q .body)

# Check mergeability and conflicts
mergeable=$(gh pr view --json mergeable -q .mergeable)

if [ "$mergeable" = "CONFLICTING" ]; then
  # Run the `pull` skill to handle fetch + merge + conflict resolution.
  # Then run the `push` skill to publish the updated branch.
fi

# Watch checks
if ! gh pr checks --watch; then
  gh pr checks
  exit 1
fi

# Squash-merge
gh pr merge --squash --subject "$pr_title" --body "$pr_body"
```

## Failure Handling

- If checks fail, pull details with `gh pr checks`, then fix locally, commit
  with the `commit` skill, push with the `push` skill, and re-run checks.
- Use judgment to identify flaky failures. If a failure is a flake, you may
  proceed without fixing it.
- If mergeability is `UNKNOWN`, wait and re-check.
- Do not merge while review comments are outstanding.

## Review Handling

- Review comments must be acknowledged before merge.
- Fetch review comments via `gh api` and reply.
- All GitHub comments generated by this agent must be prefixed with `[opencode]`.
